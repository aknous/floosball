## Tier 1 â€” Operational safety (cheap, outsized downside if skipped)

### â¬ś 1.1 No off-box DB backup â€” HIGHEST real risk to user data
`floosball.db` **and** all offseason snapshots live on a single Fly volume.
Volume loss/corruption = every account, card, and currency balance gone. Nothing
replicates off-box. (The offseason snapshots are *not* backups â€” same volume,
offseason-only.)
**Fix:** Litestream sidecar (continuous WAL â†’ S3/R2/Tigris) *or* a daily
`VACUUM INTO` upload. Litestream also gives point-in-time restore and could later
host the offseason rollback (Â§2.4). **Effort: S.**

### â¬ś 1.2 Sim task dies silently; `/health` can't tell âś…verified
`run_api.py:325` `asyncio.create_task(runSimulation())` has no `add_done_callback`,
and the `while True:` season loop (`floosballApplication.py:344`) has no try/except
around `runSeasonSimulation()` / `handleOffseason()`. An exception in
week-rollover / playoffs / awards / offseason kills the sim while the API serves
stale data â€” and `/health` (`api/main.py:68`) returns static `{"status":"ok"}`, so
Fly never restarts it. (Individual *games* are protected; the loop around them is
not.)
**Fix:** done-callback that logs `task.exception()` + re-raises so Fly restarts
(resume logic recovers); wrap the loop body; make `/health` assert sim liveness
(last-progress timestamp). **Effort: S.**

### â¬ś 1.3 No error tracking; app log is ephemeral
No Sentry anywhere. `logs/floosball.log` is on the container FS (wiped each
deploy); game crash dumps survive (`/data/crashes`) but a *sim-killing* error
(Â§1.2) vanishes. ~76 `except: pass` sites hide failures.
**Fix:** add Sentry (one-line FastAPI integration); point the file log at
`/data/logs`; downgrade the silent excepts to at least `logger.debug`. **Effort: S.**

### âś”ď¸Ź 1.4 Offseason snapshot growth â€” DONE (`feature/offseason-snapshot-slim`, a541ebf)
`_snapshotDbForPhase` copied the full DB (57MB at S14) on **every** phase
transition (~6), but the restore only ever rolls back 3 phases
(`rookie_draft`, `fa_draft`, `training`). ~90% of each copy was in-season history
(`game_player_stats` 20.5MB, `weekly_card_bonuses` 16.3MB, `pick_em_picks` 3MBâ€¦)
that drafts never touch â€” exactly the part that grows every season.
**Shipped:** (1) only snapshot the 3 rollback-able phases; (2) table-scope the
snapshot to the offseason-mutable tables (exclude the large in-season append-only
tables); restore now does ATTACH + per-table replace instead of a file swap.
~342MB/season â†’ ~45MB/season, flat across seasons. Validated by
`test_offseason_snapshot.py` + a fast offseason sim. See Â§2.4 for the longer-term
idempotency option that would remove snapshots entirely.

### âś”ď¸Ź 1.5 Secrets in `config.json` â€” resolved (owner) âś…verified
`config.json` is git-tracked with a non-empty `adminPassword` (len 9) and
`botApiKey` (len 20). **Owner:** these are *local* values; prod admin password is
a `fly secret` (env-override path in `config_manager.py` already prefers env).
**Remaining check:** confirm `BOT_API_KEY` is also a fly secret (if the prod bot
still uses the committed key, that one's live). Optional hygiene: blank the file
fields + add a `config.example.json`.

### âŹ¸ 1.6 `FRESH_START` env var â€” by design (owner)
`run_api.py:95` honors `FRESH_START=1` on every boot. **Owner:** intentional
local-dev affordance; prod only ever fresh-starts via the self-clearing

## Tier 3 â€” Architectural debt (real, only when there's runway)

### â¬ś 3.1 Three god-files + the `floosball_app` global
`api/main.py` (11.1k lines, **144 routes, zero `APIRouter`s**), `floosball_game.py`
(9.5k), `seasonManager.py` (9.6k). Highest-leverage + lowest-risk is splitting

`app.state` / a `Depends` accessor first, then the router split unlocks.
**Effort: L** (incremental, one router per commit).

### â¬ś 3.2 DB session handling
~113 hand-rolled `get_session()` + `try/finally` blocks in `api/main.py`; the
`session_scope()` helper (`connection.py:2145`) is used zero times. Migrate to a
`Depends(getDb)` yield-dependency (roll out with Â§3.1). Also: one **cross-thread
commit** âś… (`api/main.py:2857` commits the sim's shared session from a request

### â¬ś 3.3 Carve broadcast + serialization out of `floosball_game.py`'s `Game`

## Service & infrastructure verdict

The single-instance / SQLite / in-process-sim / custom-WS stack is **coherent and
appropriate** for an indie live app â€” the pieces all assume one machine, which
keeps the ops burden low. **Don't over-engineer:** Postgres, a Redis WS backplane,
and a separate sim worker are a *package deal* you only need the day real
concurrent load forces horizontal scaling â€” not before.

| Service | Verdict |
|---|---|
| **SQLite** (WAL + `busy_timeout=30000`) | Keep â€” well-tuned; Postgres is premature |
| **Fly.io** (1 machine + volume) | Keep â€” but see Â§1.1 backups |
| **Vercel** (FE auto-deploy) | Keep â€” FE main â†’ prod is a release; add a CI gate (Â§2.6) |
| **Clerk** | âś”ď¸Ź Owner on **prod instance**. Repo `.env.production` still references the dev instance (`capable-dragon-13...clerk.accounts.dev`) â€” scrub the stale value so it stops misleading. |
| **Resend** | âś”ď¸Ź Owner on **paid tier** â€” the 100/day free-cap concern is resolved. (Docs say "AWS SES" â€” stale; it's Resend.) |
| **Discord bot** | Keep â€” separate Node service is the right call. |
| **In-process sim** | Keep, but harden: offload fast/catch-up batches via `asyncio.to_thread` so CPU-bound sim doesn't starve the event loop + health checks. **Effort: M.** |
