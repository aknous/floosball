# Floosball Architecture & Improvement Audit вЂ” 2026-05-29

A whole-project audit (backend + frontend) for forward-looking improvements:
architecture, legacy cruft, documentation drift, and service/infra fit. The app
is live in prod with minimal issues вЂ” this is a **backlog of improvements**, not
a bug list. Built from scratch then iteratively refactored, so debt is real but
concentrated.

**Method:** seven parallel investigators across the major dimensions; the
high-stakes, action-driving claims were then verified by hand (marked вњ… below).

**Verdict:** the bones are sound (clean coordinator/DI/repository patterns,
WAL-tuned SQLite, real crash-recovery + offseason-resume). The debt is
concentrated in three oversized files, one global everything reaches through,
legacy islands from the TS migration, and stale docs. **The real risks are
operational, not architectural** вЂ” the architecture won't lose data; the missing
ops scaffolding can.

## Tier 1 вЂ” Operational safety (cheap, outsized downside if skipped)

### рџ”§ 1.1 Off-box DB backup вЂ” PARTIAL (`feature/ops-safety`, e827fa0)
`floosball.db` **and** all offseason snapshots live on a single Fly volume.
Volume loss/corruption = every account, card, and currency balance gone.
**Shipped:** periodic consistent `VACUUM INTO` backups with rotation
(`backup_manager.py`), atomic write, env-gated (`BACKUP_*`). Off the *primary*
volume when `BACKUP_DIR` points at a second Fly volume.
**Remaining (true off-box):** Litestream sidecar (continuous WAL в†’ S3/R2/Tigris)
for point-in-time restore вЂ” needs a bucket + creds + a test deploy. Could later
also host the offseason rollback (В§2.4).

### вњ”пёЏ 1.2 Sim task supervision + sim-liveness /health вЂ” DONE (e827fa0) вњ…verified
The background season loop had no done-callback and no try/except, so an
exception in week-rollover/playoffs/awards/offseason silently killed the sim
while the API served a frozen season and `/health` returned static `ok`.
**Shipped:** the sim task is supervised вЂ” unhandled exit logs CRITICAL (+ Sentry)
then `os._exit(1)` so Fly restarts and the resume logic recovers; a cancelled
task (shutdown) is ignored. `/health` now returns 503 when the sim task is
done/crashed (checks `floosball_app.simTask`). Removed a shadowed duplicate
`/health` route. (A *frozen* sim вЂ” a hang, not an exception вЂ” isn't covered; a
heartbeat could be added later if it ever matters.)

### вњ”пёЏ 1.3 Error tracking (Sentry) вЂ” DONE (e827fa0); log-path TODO
No Sentry; `logs/floosball.log` is on the ephemeral container FS.
**Shipped:** `_initSentry()` (sentry-sdk, env-gated on `SENTRY_DSN`, no-op when
unset or uninstalled); the sim supervisor reports fatal exceptions explicitly.
**ACTION: set the `SENTRY_DSN` fly secret to turn it on.**
**Remaining:** point the file log at `/data/logs` so it survives restarts;
downgrade the ~76 silent `except: pass` sites to `logger.debug`.

### вњ”пёЏ 1.4 Offseason snapshot growth вЂ” DONE (`feature/offseason-snapshot-slim`, a541ebf)
`_snapshotDbForPhase` copied the full DB (57MB at S14) on **every** phase
transition (~6), but the restore only ever rolls back 3 phases
(`rookie_draft`, `fa_draft`, `training`). ~90% of each copy was in-season history
(`game_player_stats` 20.5MB, `weekly_card_bonuses` 16.3MB, `pick_em_picks` 3MBвЂ¦)
that drafts never touch вЂ” exactly the part that grows every season.
**Shipped:** (1) only snapshot the 3 rollback-able phases; (2) table-scope the
snapshot to the offseason-mutable tables (exclude the large in-season append-only
tables); restore now does ATTACH + per-table replace instead of a file swap.
~342MB/season в†’ ~45MB/season, flat across seasons. Validated by
`test_offseason_snapshot.py` + a fast offseason sim. See В§2.4 for the longer-term
idempotency option that would remove snapshots entirely.

### вњ”пёЏ 1.5 Secrets in `config.json` вЂ” resolved (owner) вњ…verified
`config.json` is git-tracked with a non-empty `adminPassword` (len 9) and
`botApiKey` (len 20). **Owner:** these are *local* values; prod admin password is
a `fly secret` (env-override path in `config_manager.py` already prefers env).
**Remaining check:** confirm `BOT_API_KEY` is also a fly secret (if the prod bot
still uses the committed key, that one's live). Optional hygiene: blank the file
fields + add a `config.example.json`.

### вЏё 1.6 `FRESH_START` env var вЂ” by design (owner)
`run_api.py:95` honors `FRESH_START=1` on every boot. **Owner:** intentional
local-dev affordance; prod only ever fresh-starts via the self-clearing
`/data/.fresh` flag file. Not a prod risk. *Optional* hardening: ignore the env
var when `TIMING_MODE == scheduled`.

## Tier 2 вЂ” Cleanup (safe, mostly mechanical)

### в¬њ 2.1 Alembic is broken *and* unused вЂ” retire it вњ…verified
Duplicate revision IDs (`a1b2c3d4e5f6`, `d4e5f6g7h8i9` each defined twice) and a
4-way branch off `z1a2b3c4d5e6` вЂ” `alembic upgrade head` would error today. The

**Fix:** formally retire alembic (move `alembic/` to `legacy/`, document inline

### в¬њ 2.4 (covered by В§1.4) вЂ” longer-term: idempotent offseason steps
Once the snapshot slimming ships, the eventual ideal is to make draft/training

**Fix:** gate behind `SIM_ANALYTICS=1`, default off in prod. **Effort: S.**

### в¬њ 2.6 No CI gate
No GitHub Actions in either repo. Frontend builds with `CI=false` (`vercel.json`),
so warnings/type errors ship; FE main auto-deploys to Vercel with no gate.
**Fix:** one PR workflow per repo вЂ” backend `pytest` + `import api.main` smoke;
frontend `tsc --noEmit` + `npm run build`. **Effort: Sв†’M.**

## Tier 3 вЂ” Architectural debt (real, only when there's runway)

### в¬њ 3.1 Three god-files + the `floosball_app` global
`api/main.py` (11.1k lines, **144 routes, zero `APIRouter`s**), `floosball_game.py`
(9.5k), `seasonManager.py` (9.6k). Highest-leverage + lowest-risk is splitting
`main.py` into routers (comment-banner sections are already the boundaries; paths
stay identical, so the frontend is untouched). **Blocker:** the `floosball_app`
module global, which handlers, `api_response_builders.py`, *and* `seasonManager`
(`from api.main import _computeLeaderboardData`) all reach through вЂ” move it to
`app.state` / a `Depends` accessor first, then the router split unlocks.
**Effort: L** (incremental, one router per commit).

### в¬њ 3.2 DB session handling
~113 hand-rolled `get_session()` + `try/finally` blocks in `api/main.py`; the
`session_scope()` helper (`connection.py:2145`) is used zero times. Migrate to a
`Depends(getDb)` yield-dependency (roll out with В§3.1). Also: one **cross-thread
commit** вњ… (`api/main.py:2857` commits the sim's shared session from a request
handler вЂ” admin-only, low-frequency, latent corruption vector; the endpoint
should open its own session). Long-lived shared session
(`floosballApplication.py:35`) with default `expire_on_commit=True` вЂ” fine at this
scale; watch WAL/memory on long uptimes. FKs are declared but **not enforced**
(`foreign_keys` pragma off) вЂ” document it. **Effort: M.**

### в¬њ 3.3 Carve broadcast + serialization out of `floosball_game.py`'s `Game`
The `Game` class owns simulation, clock, WP/EP math, **broadcasting** (imports
`api.game_broadcaster`), **serialization** (`getGameData`, `_buildGameStatsSnapshot`,
`_collect_*`), direct DB access, and analytics. Extract a `GameSerializer` and
invert the broadcast dependency (emit to an injected sink) so the sim layer stops
importing the API layer. Keep the physics/play-calling core intact. **Effort: L.**

### в¬њ 3.4 Split `seasonManager.py` by domain
Conflates season loop, scheduling, playoffs, offseason state machine, card-effect
settlement (`_processWeekCardEffects` ~800 lines), economy/funding/prizes, pick-em
resolution, emails, GM votes. Extract the independent domains first (scheduling,
card settlement, economy); leave the offseason state machine for last. **Effort: L.**

### в¬њ 3.5 Frontend data layer
180 raw `fetch()` calls across 47 files; the typed `services/api.ts` is imported by
**1** file; auth-token plumbing copy-pasted ~50Г—. No caching/dedup. Phase it:
authed-fetch wrapper first (quick win), then TanStack Query (incremental;
invalidate on WS events). **Effort: L.**

### в¬њ 3.6 Frontend safety + type drift
**No error boundary anywhere** вЂ” one bad WS payload white-screens the whole app
(~40 lines to fix; do this soon, it's S/high-value). **15 tsc errors** the build
doesn't catch (`config-overrides.js` strips the TS plugin) вЂ” mostly `EventType`
union drift in `types/websocket.ts` + `CardData` shape drift; fix + add a
`typecheck` script. **Effort: SвЂ“M.**

### в¬њ 3.7 CRA в†’ Vite
`react-app-rewired` patches around a deprecated toolchain; no code-splitting
(`React.lazy` unused) в†’ 1.6MB bundle ships AdminPage (2.4k lines) to everyone.
Migrate to Vite + route-level lazy loading. Next.js not warranted. **Effort: M**
(1вЂ“2 days; `REACT_APP_*` в†’ `VITE_*`).

### в¬њ 3.8 Tests
Of 10 `test_*.py`, only 2 assert anything and **both are currently broken**
(`test_momentum.py` stale constant `MOMENTUM_NEUTRAL_ZONE`; `test_4th_down.py`
stale `gameRules` mock); the rest just print. None are pytest-shaped. Money/card/
fantasy paths вЂ” the consequential ones вЂ” are untested. Repair the two, wire into
CI (В§2.6), then add characterization tests for currency + fantasy/card scoring.
**Effort: Mв†’L.**

## Service & infrastructure verdict

The single-instance / SQLite / in-process-sim / custom-WS stack is **coherent and
appropriate** for an indie live app вЂ” the pieces all assume one machine, which
keeps the ops burden low. **Don't over-engineer:** Postgres, a Redis WS backplane,
and a separate sim worker are a *package deal* you only need the day real
concurrent load forces horizontal scaling вЂ” not before.

| Service | Verdict |
|---|---|
| **SQLite** (WAL + `busy_timeout=30000`) | Keep вЂ” well-tuned; Postgres is premature |
| **Fly.io** (1 machine + volume) | Keep вЂ” but see В§1.1 backups |
| **Vercel** (FE auto-deploy) | Keep вЂ” FE main в†’ prod is a release; add a CI gate (В§2.6) |
| **Clerk** | вњ”пёЏ Owner on **prod instance**. Repo `.env.production` still references the dev instance (`capable-dragon-13...clerk.accounts.dev`) вЂ” scrub the stale value so it stops misleading. |
| **Resend** | вњ”пёЏ Owner on **paid tier** вЂ” the 100/day free-cap concern is resolved. (Docs say "AWS SES" вЂ” stale; it's Resend.) |
| **Discord bot** | Keep вЂ” separate Node service is the right call. |
| **In-process sim** | Keep, but harden: offload fast/catch-up batches via `asyncio.to_thread` so CPU-bound sim doesn't starve the event loop + health checks. **Effort: M.** |
